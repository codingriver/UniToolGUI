## 1. 问题全景分析

### 1.1 全球同服聊天的字体挑战

全球同服聊天意味着同一屏幕可能同时出现多种语言：

```
用户A（中国）：    帮我看看这个配置
用户B（日本）：    設定を確認しました
用户C（韩国）：    확인했습니다
用户D（阿拉伯）：  تم التحقق
用户E（泰国）：    ตรวจสอบแล้ว
用户F（俄罗斯）：  Проверено
```

任何方案都需要在以下维度取得平衡：

| 维度 | 极端A（全量） | 极端B（最小） |
|------|-------------|-------------|
| 字符覆盖度 | 所有 Unicode 字符 | 仅当前语言 |
| 显存占用 | 数百 MB | 几 MB |
| 启动速度 | 慢（预加载） | 快（按需） |
| 运行流畅度 | 极流畅 | 动态生成有卡顿 |
| 包体/流量 | 大 | 小 |

### 1.2 字体内存构成分析

```
字体总内存 = 字体文件内存 + Atlas 贴图显存 + 字形元数据

1. 字体文件内存：
   内置 OTF 文件约 18MB，加载后常驻内存
   系统字体通过文件映射按需读取，内存开销较小

2. Atlas 贴图显存（核心控制目标）：
   R8 格式（灰度）： Width x Height x 1 字节
   RGBA32 格式：    Width x Height x 4 字节
   CJK 应强制使用 R8，节省 75% 显存

   尺寸参考：
   256x256  R8 = 0.06MB  | 512x512  R8 = 0.25MB
   1024x1024 R8 = 1MB    | 2048x2048 R8 = 4MB
   4096x4096 R8 = 16MB   | 4096x4096 RGBA32 = 64MB

3. 字形容量估算公式：
   单 Atlas 字形数 = (atlasSize / (samplingPointSize + padding*2))^2
   示例：1024px, size=24, padding=3 -> (1024/30)^2 ≈ 1170 字形
   示例：4096px, size=48, padding=9 -> (4096/66)^2 ≈ 3847 字形

4. 字形元数据：每字形约 200~400 字节，10000 字形约 2~4MB（可忽略）
```

### 1.3 卡顿的四个根源

| 根源 | 触发条件 | 典型耗时 | 发生频率 |
|------|---------|---------|----------|
| 首次渲染新字符 | Dynamic 模式遇未缓存字符 | 0.1~2ms/字 | 高频（聊天场景） |
| MultiAtlas 扩展 | Atlas 空间不足时新建贴图 | 5~30ms | 低频（各触发一次） |
| Fallback 链遍历失败 | 字符不在任何字体中 | 0.01~0.1ms/字 | 中频 |
| 字体文件 IO 阻塞 | new Font(path) 主线程同步读文件 | 50~500ms | 启动时一次 |

## 2. 方案一：纯静态预烘焙 Atlas

**核心思路**：针对每种目标语言，离线预生成包含该语言全量字符的静态 FontAsset。

### 内存与包体估算

| 语言 | 字符数 | Atlas 尺寸 | R8 显存 | 包体 |
|------|--------|-----------|--------|------|
| 简体中文 | ~7000 | 4096x4096 | 16MB | ~20MB |
| 日文 | ~7000 | 4096x4096 | 16MB | ~20MB |
| 韩文 | ~11000 | 4096x4096 x2 | 32MB | ~35MB |
| 拉丁扩展 | ~1000 | 1024x1024 | 1MB | ~2MB |
| 阿拉伯文 | ~1200 | 2048x2048 | 4MB | ~5MB |
| 泰文 | ~200 | 512x512 | 0.25MB | ~1MB |
| **合计** | | | **69MB** | **~83MB** |

单语言运行时只加载对应语言，显存约 17MB（中文+拉丁）。

### 优缺点

**优点：**
- 运行时零动态开销，渲染绝对流畅
- 内存完全可控，只加载当前语言
- 字形质量最高（离线高精度烘焙 SDF32）
- 无网络依赖

**缺点：**
- 无法处理生僻字、其他语种混入（聊天场景致命缺陷）
- 包体增加约 83MB 字体资产
- 全球同服聊天多语言混显时大量方块
- 新增语言需重新生成并发版

> **评价**：适合单语言固定文案 UI（菜单/按钮），完全不适合全球同服聊天场景。

## 3. 方案二：动态系统字体回退（当前方案）

**核心思路**：运行时从操作系统读取系统字体创建动态 FontAsset，按需生成字形。

### 内存特征

```
启动时：<10MB（空 Atlas + 字体文件映射对象）

稳定态（显存）：
  1000 常用汉字显示后：约 1~2MB（2~4 个 512x512 Atlas）
  5000 汉字（含生僻字）：约 4~8MB
  长期运行 + 大量聊天：理论上无上限，需监控
```

### 优缺点

**优点：**
- 支持操作系统内置的全部字符（含生僻字）
- 启动极快，Atlas 初始为空
- 零包体（使用系统字体）
- 内存按需增长，不突然跳变

**缺点：**
- 依赖目标机器系统字体（精简版 Windows 可能缺字体）
- 首次渲染新字符有 0.1~2ms 卡顿（聊天高频场景可感知）
- 不同设备字体外观不一致
- iOS/Android 沙盒字体路径不稳定
- 字符永久留在 Atlas，长期运行内存增长无法自动回收

> **评价**：桌面工具类应用的最佳选择。全球同服聊天场景下，
> 需配合方案四（聊天专用字体）解决多语言混显和卡顿问题。

## 4. 方案三：分层字体 + 按需下载

**核心思路**：字体资产按语言分包，随包内置核心语言，其余从 CDN 按需下载。

### 分包架构

| Bundle | 内容 | 大小 | 策略 |
|--------|------|------|------|
| fonts-latin | 拉丁+标点 | ~2MB | 随包内置 |
| fonts-zh-cn | 简体中文 | ~20MB | 随包内置 |
| fonts-ja | 日文 | ~20MB | 按需下载 |
| fonts-ko | 韩文 | ~35MB | 按需下载 |
| fonts-arabic | 阿拉伯文 | ~5MB | 按需下载 |
| fonts-thai | 泰文 | ~3MB | 按需下载 |
| fonts-cjk-ext | 生僻字扩展 | ~40MB | 按需下载 |

下载时机：启动检测系统语言预下载 → 进入聊天检测在线玩家语言预下载 → 遇方块字符静默触发下载。

**优点：** 包体可控、字形质量高、可热更新
**缺点：** 需 CDN 基础设施、离线场景受限、实现复杂度高

> **评价**：大型多语言游戏标准方案，需完善后端支撑。

## 5. 方案四：聊天专用轻量动态字体

**核心思路**：聊天系统独立一套轻量动态 FontAsset，与主 UI 隔离，独立管理生命周期。

### 参数对比

| 用途 | samplingPointSize | padding | 512x512 可容字形数 | 显存/Atlas |
|------|-----------------|---------|-----------------|----------|
| 主 UI | 36 | 6 | ~145 | 0.25MB |
| 聊天轻量 | 20 | 3 | ~484 | 0.25MB |
| 极限轻量 | 16 | 2 | ~820 | 0.25MB |

聊天用 samplingPointSize=20 时，覆盖 3000 常用汉字约需 6~7 张 Atlas，共约 1.75MB 显存。

### 生命周期管理代码

```csharp
public class ChatFontManager : MonoBehaviour
{
    private FontAsset _chatFont;
    private int _msgCount = 0;
    private const int REBUILD_THRESHOLD = 5000;

    public FontAsset GetChatFont()
    {
        if (_chatFont == null) _chatFont = CreateLightFont();
        return _chatFont;
    }

    public void OnMessageReceived()
    {
        if (++_msgCount > REBUILD_THRESHOLD)
        {
            _chatFont?.ClearFontAssetData(setAtlasSizeToZero: false);
            _msgCount = 0;
            Debug.Log("[ChatFont] Atlas cleared");
        }
    }

    private FontAsset CreateLightFont()
    {
        var font = new Font(SystemFontPath());
        return FontAsset.CreateFontAsset(font,
            samplingPointSize: 20, atlasPadding: 3,
            renderMode: GlyphRenderMode.SDFAA,
            atlasWidth: 512, atlasHeight: 512,
            atlasPopulationMode: AtlasPopulationMode.Dynamic,
            enableMultiAtlasSupport: true);
    }

    private string SystemFontPath()
    {
#if UNITY_EDITOR_WIN || UNITY_STANDALONE_WIN
        return @"C:\Windows\Fonts\msyh.ttc";
#elif UNITY_EDITOR_OSX || UNITY_STANDALONE_OSX
        return "/System/Library/Fonts/PingFang.ttc";
#else
        return "/system/fonts/NotoSansCJK-Regular.ttc";
#endif
    }
}
```

**优点：** 与主 UI 完全隔离、轻量参数降低显存、可独立清理重建、支持任意字符
**缺点：** 字形质量略低（samplingPointSize 较小）、首次渲染微小卡顿、需维护生命周期逻辑

> **评价**：全球同服聊天的关键优化手段，强烈推荐配合其他方案使用。

## 6. 方案五：混合分级策略（推荐）

**核心思路**：主 UI 静态高质量 + 聊天轻量动态 + 系统字体兜底，三层覆盖所有场景。

### 三层架构

```
第一层：主 UI 静态字体（高质量）
  用途：按钮/标签/标题等固定文案
  参数：Static FontAsset, SDF32, samplingPointSize=48, Atlas=4096x4096 R8
  显存：约 16MB，固定不变
  覆盖：常用汉字 7000+，覆盖 99% UI 固定文案

第二层：聊天动态字体（轻量）
  用途：聊天气泡、玩家昵称、动态消息
  参数：Dynamic FontAsset, SDFAA, samplingPointSize=20, Atlas=512x512 R8
  显存：按需增长，3000 常用字约 1.75MB，可定期清理
  覆盖：绝大多数常用字符（含多语言）

第三层：系统字体兜底（生僻字）
  用途：兜底生僻字、特殊符号、小语种
  参数：Dynamic FontAsset from 系统字体，注入 Fallback 链末尾
  显存：极少触发，可忽略
  覆盖：系统支持的全部字符
```

### 显存预算

| 层级 | 显存 | 备注 |
|------|------|------|
| 第一层主 UI Static | 16MB | 固定 |
| 第二层聊天 Dynamic | ~2MB | 稳定后 |
| 第三层系统兜底 | ~1MB | 极少触发 |
| 拉丁回退 Static | 1MB | 固定 |
| **合计** | **~20MB** | 可接受 |

> **我的建议**：方案五是全球同服聊天场景的最优解。总显存约 20MB，
> 远低于全量静态方案（69MB），覆盖度更广，聊天内存可控可清理。

## 7. Atlas 内存/显存控制深度指南

### 7.1 强制使用 R8 格式（最重要优化）

Font Asset Creator 默认 R8（灰度）格式。4096x4096 R8 = 16MB，RGBA32 = 64MB，节省 75%。
Emoji 字体例外，必须 RGBA32（彩色）。

### 7.2 samplingPointSize 对容量的影响（1024x1024 Atlas）

| samplingPointSize | padding | 可容字形数 | 推荐场景 |
|-----------------|---------|----------|---------|
| 16 | 2 | ~3200 | 极限轻量聊天 |
| 20 | 3 | ~1170 | 聊天轻量（推荐） |
| 28 | 4 | ~625 | 普通 UI |
| 36 | 6 | ~330 | 主 UI 标准 |
| 48 | 9 | ~165 | 高质量标题 |

### 7.3 MultiAtlas 上限监控

```csharp
void Update()
{
    if (_chatFont == null) return;
    int count = _chatFont.atlasTextures?.Length ?? 0;
    if (count > 8) // 8张512x512 R8 = 2MB上限
    {
        _chatFont.ClearFontAssetData(setAtlasSizeToZero: false);
        Debug.Log($"[Atlas] 超出上限({count}), 已清理");
    }
}
```

### 7.4 字体文件异步加载（消除启动 IO 卡顿）

new Font(path) 是同步 IO，大字体文件可造成 100~500ms 主线程阻塞。
解决方案：使用 UnityMainThreadDispatcher 将 IO 移到后台线程，FontAsset.CreateFontAsset 仍在主线程调用。

## 8. 渲染卡顿根因与消除方案

### 8.1 聊天预热高频汉字

```csharp
private const string COMMON_CJK =
    "的一是在不了有和人这中大为上个国我以要他时来用们生到作地" +
    "于出就分对成会可主发年动同工也能下过子说产种面而方后多定";

public void PrewarmChatFont(FontAsset font)
{
    font?.HasCharacters(COMMON_CJK, out _, true, true);
}
```

### 8.2 分帧生成（消除翻页卡顿）

```csharp
public IEnumerator PrewarmAsync(FontAsset font, string chars, int perFrame = 50)
{
    for (int i = 0; i < chars.Length; i += perFrame)
    {
        string batch = chars.Substring(i, Mathf.Min(perFrame, chars.Length - i));
        font.HasCharacters(batch, out _, true, true);
        yield return null;
    }
}
```

### 8.3 Fallback 链长度控制

- 主 UI 字体 Fallback 链 <= 3 个
- 聊天字体 Fallback 链 <= 2 个
- 高频字体放链首，系统字体兜底放链尾
- 合并同语系字体（思源黑体 SC 覆盖简繁日韩，减少链长）

## 10. 本项目落地建议

### 当前项目（CFST 桌面工具）

当前方案二已完全满足需求：
- Language: ChineseSimplified + Latin 回退
- AtlasSize: Small_512 + MultiAtlas
- RenderMode: SDFAA, samplingPointSize: 36

### 若未来增加全球同服聊天功能

**推荐迁移到方案五（混合分级）**，分三步实施：

```
Step 1：主 UI 升级为 Static FontAsset
  思源黑体 SC 生成 GB2312 全集静态 FontAsset
  Atlas: 4096x4096 R8, SDF32, samplingPointSize=48

Step 2：添加聊天专用 FontAsset
  samplingPointSize=20, padding=3, SDFAA
  Atlas=512x512, MultiAtlas 上限 8 张
  聊天组件通过 ChatFontManager 获取专属字体

Step 3：保留系统字体回退（SystemFontFallback.cs）
  注入 Fallback 链末尾，兜底生僻字和小语种
  聊天窗口打开时 PrewarmAsync 分帧预热 1000 高频汉字
```

### 字体授权注意事项

| 字体 | 授权 | 可随包分发 | 商业可用 |
|------|------|----------|---------|
| 思源黑体 SC | OFL 1.1 | ✅ | ✅ |
| Noto Sans SC | OFL 1.1 | ✅ | ✅ |
| 微软雅黑 | 商业授权 | ❌ 仅系统自带 | ⚠️ |
| 苹方 | 苹果专有 | ❌ | ❌ |

SystemFontFallback.cs 运行时读取系统字体属于「运行时使用」而非「分发字体文件」，大多数平台授权允许。

## 总结

全球化同服聊天字体无银弹，核心原则：

1. **分离关注点**：主 UI 字体与聊天字体使用独立 FontAsset，互不干扰
2. **按需分级**：固定文案用静态 Atlas，动态内容用动态 Atlas
3. **R8 格式**：CJK 字体必须 R8，节省 75% 显存
4. **预热优先**：聊天场景预热高频字符，消除首字渲染卡顿
5. **生命周期管理**：聊天字体定期清理，防止长期运行内存增长
6. **系统字体兜底**：最后一道防线，任何字符不显示方块

对于本项目 CFST，当前方案已足够。若扩展全球同服产品，
推荐按方案五渐进实施，优先保证多语言显示正确，再逐步优化性能和内存。

## 11. Atlas 事件监控系统

### 11.1 TextCore 字形生成事件

Unity TextCore 在动态添加字形时触发静态事件，可用于监控 Atlas 变化：

```csharp
void OnEnable()
{
    // 字形写入 Atlas 时触发（每次 Atlas 贴图内容变化）
    FontAsset.OnFontAssetTextureChanged += OnAtlasChanged;
}
void OnDisable()
{
    FontAsset.OnFontAssetTextureChanged -= OnAtlasChanged;
}

private void OnAtlasChanged(FontAsset fa)
{
    int atlasCount = fa.atlasTextures?.Length ?? 0;
    int charCount  = fa.characterTable?.Count  ?? 0;
    var tex = fa.atlasTexture;
    float vramMB = tex != null
        ? atlasCount * tex.width * tex.height / (1024f * 1024f) : 0;

    Debug.Log($"[Atlas] {fa.name} | 张数={atlasCount} | 字形={charCount} | 显存={vramMB:F2}MB");

    if (atlasCount > 8) // 超出上限，触发清理
        fa.ClearFontAssetData(setAtlasSizeToZero: false);
}
```

### 11.2 缺失字符检测

```csharp
// 在设置文本前主动检查，缺失字符触发回调
public void CheckMissing(string text, FontAsset fa, System.Action<char> onMissing)
{
    foreach (char c in text)
    {
        if (c < 32) continue;
        if (!fa.HasCharacter(c, searchFallbacks: true))
            onMissing?.Invoke(c);
    }
}
```

### 11.3 Atlas 使用率统计

```csharp
public struct AtlasStats
{
    public int   atlasCount; // Atlas 张数
    public int   charCount;  // 已缓存字形数
    public float vramMB;     // 估算显存
}

public static AtlasStats GetStats(FontAsset fa)
{
    if (fa == null) return default;
    var tex = fa.atlasTexture;
    return new AtlasStats
    {
        atlasCount = fa.atlasTextures?.Length ?? 0,
        charCount  = fa.characterTable?.Count  ?? 0,
        vramMB     = tex != null
            ? (fa.atlasTextures.Length * tex.width * tex.height) / (1024f * 1024f) : 0
    };
}
```

## 12. 异步字体加载全流程

### 12.1 为什么必须异步

```
new Font(path) 同步耗时参考（主线程）：
  Arial 0.3MB    : 5~20ms
  微软雅黑 18MB  : 100~500ms  ← 明显掉帧
  思源黑体 7MB   : 50~200ms

超过 16ms 掉帧，超过 100ms 用户可感知。
解决：IO 放后台线程，FontAsset.CreateFontAsset 必须回主线程执行。
```

### 12.2 async/await 版本

```csharp
public async Task<FontAsset> LoadAsync(string path,
    int samplingPointSize=20, int padding=3, int atlasSize=512)
{
    // Step1: 后台线程 IO
    Font font = await Task.Run(() =>
    {
        if (!File.Exists(path)) return null;
        try { return new Font(path); } catch { return null; }
    });
    if (font == null) return null;

    // Step2: 主线程创建 FontAsset（Unity 要求）
    return FontAsset.CreateFontAsset(font,
        samplingPointSize, padding, GlyphRenderMode.SDFAA,
        atlasSize, atlasSize, AtlasPopulationMode.Dynamic, true);
}
```

### 12.3 协程版本（兼容旧写法）

```csharp
public IEnumerator LoadCoroutine(string path, Action<FontAsset> cb,
    int samplingPointSize=20, int padding=3, int atlasSize=512)
{
    Font font = null; bool done = false;
    ThreadPool.QueueUserWorkItem(_ =>
    {
        try { font = File.Exists(path) ? new Font(path) : null; }
        finally { done = true; }
    });
    yield return new WaitUntil(() => done);
    if (font == null) { cb?.Invoke(null); yield break; }
    var fa = FontAsset.CreateFontAsset(font,
        samplingPointSize, padding, GlyphRenderMode.SDFAA,
        atlasSize, atlasSize, AtlasPopulationMode.Dynamic, true);
    fa.name = "[Async]" + Path.GetFileNameWithoutExtension(path);
    cb?.Invoke(fa);
}
```

### 12.4 注意事项

```
1. FontAsset.CreateFontAsset 必须在主线程调用，不能放在 Task.Run 内
2. Unity 的 async/await 默认在主线程 SynchronizationContext 恢复
   所以 await Task.Run(...) 之后自动回主线程，无需手动 dispatch
3. 协程版本需要配合 UnityMainThreadDispatcher 或直接 yield 等待
4. 多个字体路径可并行 IO，再顺序创建 FontAsset：
   var fonts = await Task.WhenAll(paths.Select(p => Task.Run(() => new Font(p))));
```

## 13. 异步动态字符添加

### 13.1 线程限制说明

```
FontAsset 字形生成链（全部必须主线程）：
  TryAddCharacters() -> FreeType 光栅化 -> SDF 计算 -> Texture2D.Apply()

无法后台线程生成字形，但可以：
  - 分帧生成（每帧处理少量字符，分散开销）
  - 场景加载期批量预热（Loading 界面期间生成）
  - 只预热高频字符（3000 常用汉字覆盖 99% 日常用字）
```

### 13.2 分帧预热（核心优化）

```csharp
// 每帧处理 charsPerFrame 个字符，避免单帧卡顿
// SDFAA 模式每字约 0.1~0.5ms，charsPerFrame=10 每帧消耗约 1~5ms
public IEnumerator PrewarmAsync(FontAsset font, string chars, int charsPerFrame=10)
{
    // 过滤已缓存字符，只处理未命中的
    var sb = new System.Text.StringBuilder();
    foreach (char c in chars)
        if (!font.HasCharacter(c, false)) sb.Append(c);
    string todo = sb.ToString();

    for (int i = 0; i < todo.Length; i += charsPerFrame)
    {
        string batch = todo.Substring(i, Mathf.Min(charsPerFrame, todo.Length - i));
        font.TryAddCharacters(batch, out string missing);
        if (!string.IsNullOrEmpty(missing))
            Debug.LogWarning($"[Prewarm] 无法生成字符: {missing}");
        yield return null; // 等待下一帧
    }
    Debug.Log($"[Prewarm] 完成，共预热 {todo.Length} 个字符");
}
```

### 13.3 聊天消息预热流程

```csharp
// 收到消息 -> 先预热字符 -> 等预热完成 -> 再显示文本
// 消除首字渲染卡顿，用户体验更流畅
IEnumerator ShowMessageAfterPrewarm(string text, Label label, FontAsset chatFont)
{
    yield return StartCoroutine(PrewarmAsync(chatFont, text, charsPerFrame: 20));
    label.text = text; // 预热完成后再赋值，不会触发动态生成
}
```

### 13.4 HasCharacter vs TryAddCharacters 区别

| API | 作用 | 是否生成字形 | 线程 |
|-----|------|------------|------|
| `HasCharacter(c, false)` | 仅查询是否已缓存 | 否 | 任意 |
| `HasCharacter(c, true)` | 查询+搜索 Fallback | 否（只查询） | 主线程 |
| `HasCharacters(str, out _, true, true)` | 查询并触发动态生成 | **是** | 主线程 |
| `TryAddCharacters(str, out missing)` | 主动添加字符到 Atlas | **是** | 主线程 |

推荐预热时使用 `TryAddCharacters`，它会返回无法生成的字符列表（missing），便于调试。

## 14. 字体清理完整指南

### 14.1 三种清理方式对比

| 方式 | API | 清理内容 | Atlas 重置 | 字形重建时机 | 适用场景 |
|------|-----|---------|-----------|------------|----------|
| 清除字形数据 | `ClearFontAssetData(false)` | 字符表+字形表 | 保留贴图尺寸 | 下次渲染触发 | 定期维护 |
| 清除+缩小Atlas | `ClearFontAssetData(true)` | 字符表+字形表+贴图 | 缩回初始尺寸 | 下次渲染触发 | 最大化释放显存 |
| 销毁 FontAsset | `Destroy(fontAsset)` | 全部资源 | 完全释放 | 需重新 Create | 语言切换/退出 |

### 14.2 ClearFontAssetData 详解

```csharp
// setAtlasSizeToZero=false：清除字形数据，Atlas 贴图保留原尺寸（推荐）
// 优点：下次渲染时直接写入现有贴图，无需重新分配显存
// 缺点：显存不释放
fontAsset.ClearFontAssetData(setAtlasSizeToZero: false);

// setAtlasSizeToZero=true：清除字形数据并将 Atlas 缩回初始尺寸
// 优点：释放 MultiAtlas 额外分配的显存
// 缺点：下次渲染需重新分配贴图，有一次性开销
fontAsset.ClearFontAssetData(setAtlasSizeToZero: true);
```

### 14.3 清理时机策略

```csharp
public class FontCleanupController : MonoBehaviour
{
    [SerializeField] private FontAsset chatFont;
    [SerializeField] private int maxAtlasCount    = 8;    // Atlas 张数上限
    [SerializeField] private int maxMsgBeforeClean = 5000; // 消息条数触发
    private int _msgCount;

    // 策略1：消息条数触发（适合聊天场景）
    public void OnMessageReceived()
    {
        if (++_msgCount >= maxMsgBeforeClean)
        {
            chatFont.ClearFontAssetData(false);
            _msgCount = 0;
            Debug.Log("[Font] 消息数触发清理");
        }
    }

    // 策略2：Atlas 张数触发（自动监控）
    void Update()
    {
        int count = chatFont?.atlasTextures?.Length ?? 0;
        if (count > maxAtlasCount)
        {
            chatFont.ClearFontAssetData(false);
            Debug.Log($"[Font] Atlas({count})超限触发清理");
        }
    }

    // 策略3：场景/页面切换时清理
    public void OnSceneUnload()
    {
        chatFont?.ClearFontAssetData(true); // 释放显存
    }

    // 策略4：应用进入后台时清理（移动端省内存）
    void OnApplicationPause(bool paused)
    {
        if (paused) chatFont?.ClearFontAssetData(true);
    }
}
```

### 14.4 完全销毁与重建

```csharp
// 语言切换时完全重建 FontAsset
public class FontRebuildManager : MonoBehaviour
{
    private FontAsset _chatFont;
    private string    _currentPath;

    public async Task SwitchLanguageFont(string newFontPath)
    {
        // Step1: 断开注入
        var ts = FindObjectOfType<UIDocument>()?.panelSettings?.textSettings;
        if (ts != null && ts.defaultFontAsset == _chatFont)
            ts.defaultFontAsset = null;

        // Step2: 从 Fallback 链移除
        var mainFont = Resources.Load<FontAsset>("Fonts/MainFont");
        mainFont?.fallbackFontAssetTable.Remove(_chatFont);

        // Step3: 销毁旧 FontAsset
        if (_chatFont != null) Destroy(_chatFont);
        _chatFont = null;

        // Step4: 异步加载新字体
        var loader = GetComponent<AsyncFontLoader>();
        _chatFont = await loader.LoadAsync(newFontPath);
        _currentPath = newFontPath;

        // Step5: 重新注入
        mainFont?.fallbackFontAssetTable.Add(_chatFont);
        Debug.Log($"[Font] 语言字体切换完成: {newFontPath}");
    }
}
```

### 14.5 清理后的首帧重建开销

```
清理后首次渲染时，所有字符重新触发动态生成：

场景：清理后立即显示 500 个汉字
  每字约 0.3ms (SDFAA) -> 500 x 0.3ms = 150ms 卡顿

解决方案：
  清理时机选在页面不可见时（Loading 界面、后台）
  清理后立即执行 PrewarmAsync 预热高频字符
  再切换页面显示，首帧无动态生成开销

推荐流程：
  OnSceneUnload -> ClearFontAssetData
  OnSceneLoad   -> PrewarmAsync（高频字符）-> 显示 UI
```

## 15. 打包与 Atlas 纹理说明

### 15.1 Dynamic vs Static 打包行为

| 模式 | Atlas 是否打包进 Build | 原因 |
|------|----------------------|------|
| **Dynamic**（当前方案） | ❌ 不打包 | Atlas 是运行时内存对象，不是资源文件 |
| **Static** | ✅ 打包 | Atlas 是预烘焙贴图，作为资源引用存储 |

本项目 `SystemFontFallback.cs` 使用 `AtlasPopulationMode.Dynamic` + `new Font(systemPath)`：
- 打包时没有任何字体 Atlas 贴图被打包，Build 包体中字体资产 = 0 字节
- 运行时从系统路径创建 FontAsset，Atlas 在内存中按需生成
- 退出应用时所有内存自动释放

### 15.2 Unity 自动清理行为

```
Editor Play Mode：
  进入 Play Mode 时 Dynamic Atlas 被清空重建（Editor 沙盒机制）
  这是正常行为，不影响构建版本

Build 运行时：
  每次启动：重新从系统字体创建空 Atlas
  正常退出：所有内存/显存自动释放（OS 回收）
  强制关闭：同上，OS 自动回收进程内存
  不存在「残留 Atlas 需要手动清理」的问题
```

### 15.3 何时才需要手动清理

| 场景 | 需要清理 | 原因 |
|------|---------|------|
| 打包（Build） | ❌ | Dynamic Atlas 不打包，启动时为空 |
| 应用正常退出 | ❌ | OS 自动回收全部内存 |
| 短期运行工具软件（<2h） | ❌ | 字符集稳定，Atlas 增长有限 |
| 长期运行（>2h）+ 大量动态文本 | ⚠️ 建议 | 防止 Atlas 持续增长占用显存 |
| 语言切换 | ⚠️ 建议 | 旧语言字形释放显存，切换更干净 |
| 聊天场景（全球同服） | ⚠️ 建议 | 大量临时字符积累，定期清理 |

### 15.4 本项目（CFST）结论

**完全不需要处理清理逻辑。**

- 界面文字固定，字符集稳定，Atlas 增长有上限
- 打包后 Dynamic Atlas 启动时为空，字符按需生成
- 退出时 OS 自动释放全部内存
- 无需在代码中添加任何清理调用
